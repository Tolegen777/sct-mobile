# App Lock (пин-код + Face ID) — дизайн

> Дата: 2026-06-13
> Статус: утверждён, готов к плану реализации

## Цель

Локальная блокировка приложения: после однократного входа телефон+пароль
пользователь при открытии приложения разблокирует его 4-значным пин-кодом или
Face ID / Touch ID — как в банковских приложениях.

**Бэкенд не задействован.** Это полностью клиентская функция поверх уже
сохранённой JWT-сессии (`expo-secure-store`, см.
`src/shared/api/token-storage.ts`).

## Решения (согласовано с пользователем)

| Вопрос | Решение |
|---|---|
| Включение | Опционально, в профиле (по умолчанию выключено) |
| Когда блокировать | Холодный старт + возврат из фона, если прошло >30с |
| Неверный пин | 5 попыток → logout + вход паролем (он же путь «забыл пин») |
| Длина пина | 4 цифры |
| Face ID | Опция поверх пина; при отказе/неудаче — ввод пина |
| Модель безопасности | Подход A: локальный гейт (соль+хэш в SecureStore), без шифрования токенов |

## Архитектура

Новый модуль `src/features/app-lock/`:

| Файл | Назначение |
|---|---|
| `store.ts` | zustand-стор: `status: 'pending' \| 'locked' \| 'unlocked'`; методы `hydrate / lock / unlock` |
| `storage.ts` | обёртка над SecureStore: `enabled`, `biometricEnabled`, `pinSalt`, `pinHash`, `failedAttempts` |
| `crypto.ts` | `generateSalt()`, `hashPin(pin, salt)` через `expo-crypto` (SHA-256) — чистые функции |
| `biometrics.ts` | обёртка над `expo-local-authentication`: `isAvailable()`, `authenticate()` |
| `config.ts` | константы: `PIN_LENGTH = 4`, `MAX_ATTEMPTS = 5`, `BACKGROUND_GRACE_MS = 30000`; чистый хелпер `shouldLockOnResume(backgroundedAt, now)` |
| `PinKeypad.tsx` | цифровая клавиатура 0–9 + удаление + индикатор-точки |
| `LockScreen.tsx` | экран замка (оверлей) |
| `AppLockGate.tsx` | обёртка в корневом layout: рисует `LockScreen` поверх всего при `status==='locked'` |

Новые зависимости: `expo-local-authentication`, `expo-crypto` (обе официальные
Expo, ставятся через `npx expo install`).

## Поток блокировки/разблокировки

1. **Старт** — в `app/_layout.tsx` после `authStore.hydrate()`:
   `appLock.hydrate()` грузит конфиг. Если `enabled && phase==='authed'` →
   `status='locked'`, иначе `status='unlocked'`. Сплеш скрываем после этого шага.
2. **Уход в фон** — отдельный `AppState` listener в модуле app-lock (внутри
   `AppLockGate` или `store`) запоминает `backgroundedAt`. Не смешиваем с
   `setupAppStateFocus` из `src/core/online-focus.ts` — там своя задача
   (react-query online manager), разные ответственности держим раздельно.
3. **Возврат** — если замок включён и `shouldLockOnResume(backgroundedAt, now)`
   (прошло >30с) → `lock()`.
4. **`status==='locked'`** — `AppLockGate` рендерит `LockScreen` поверх табов и
   всех экранов; нижняя навигация недоступна.
5. **Разблокировка** — верный пин ИЛИ успешный Face ID → `unlock()`, сброс
   `failedAttempts`.

## Экран замка (LockScreen)

- При появлении: если `biometricEnabled && isAvailable()` — сразу вызываем
  биометрию. Успех → `unlock()`. Отмена/неудача → ввод пина.
- Пин-пад: 4 точки, цифры 0–9, удаление. На 4-й цифре сверяем
  `hashPin(entered, salt) === pinHash`. Верно → `unlock()`. Неверно → анимация
  «тряски» точек, очистка, текст «осталось N попыток».
- **5 неверных** → `authStore.logout()` + `storage.clear()` → пользователь
  становится гостем, замок исчезает, открывается обычный вход паролем.
- Кнопки внизу: «Войти паролем» (= logout, путь «забыл пин») и иконка Face ID
  (если включён) для повторного вызова биометрии.

## Настройка (профиль → секция «Безопасность»)

- Тумблер «Код-пароль»:
  - ВКЛ → экран `app/app-lock/setup.tsx` (модалка): ввести пин → повторить →
    сохранить соль+хэш, `enabled=true`.
  - ВЫКЛ → подтверждение → `storage.clear()`, `enabled=false`.
- Тумблер «Face ID / Touch ID» — виден только если пин включён И
  `biometrics.isAvailable()`.
- Кнопка «Сменить код» — когда замок включён (тот же экран `setup.tsx`).

## Связь с авторизацией

- Замок активен только при `phase==='authed'`. Гостя не блокируем.
- Любой logout (ручной из профиля, по 5 попыткам, истёкший refresh) стирает
  пин и конфиг замка. Пин привязан к сессии: вышел → при следующем входе
  настраиваешь заново.

## Тестирование

В проекте нет тест-инфраструктуры (jest не настроен; проверка велась через
`tsc --noEmit` + сборка бандла Metro). Поэтому:

- Чистая логика вынесена в тестируемые функции: `hashPin`, `generateSalt`
  (`crypto.ts`), `shouldLockOnResume` (`config.ts`), счётчик попыток.
- Проверка корректности — `tsc --noEmit` + сборка бандла + ручной чеклист на
  устройстве:
  1. Включить код-пароль в профиле, задать пин.
  2. Свернуть приложение >30с → при возврате показывается замок.
  3. Свернуть <30с → замок НЕ показывается.
  4. Холодный перезапуск → замок показывается.
  5. Включить Face ID → при открытии сразу прелагается лицо; отмена → пин.
  6. Ввести неверный пин 5 раз → выход на экран входа паролем.
  7. «Войти паролем» → выход на экран входа.
  8. Выключить код-пароль → замок больше не появляется.
- Автотестов не будет, пока не поднята тест-инфра (честное ограничение).

## Вне scope (YAGNI)

- Шифрование токенов пином (подход B) — оверкилл для сервисного приложения.
- SMS-код для входа (требует бэкенд, см. `../sct-web/BACKEND_NOTES.md` §4.1).
- Нарастающие таймауты при ошибках пина — выбран простой лимит 5 попыток.
- Сохранение пина между сессиями после logout.
